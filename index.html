대용량조회 성능튜닝 운영성능개선

상황
특정 물류·재고 조회 화면에서 응답 시간이 과도하게 오래 걸린다는 문의가 접수되었고, 쿼리 분석 결과 비효율적인 SQL 구조로 인해 성능 저하가 심화된 상태였습니다.

문제 분석
스칼라 서브쿼리와 사용자 정의 함수가 다수 포함되어 있었고, IN 조건 남용과 인덱스 컬럼 변형 함수 사용으로 인덱스 미사용 구간이 발생하고 있었습니다. 또한 반복 처리와 다중 update 문 수행이 성능 저하를 가중시키고 있었습니다.

개선 및 조치
스칼라 서브쿼리와 함수를 JOIN 기반 구조로 변경하고, 조건절을 EXISTS 및 JOIN 방식으로 재검토했습니다. 인덱스 컬럼 변형 함수를 제거했으며, loop 기반 다수건 update는 임시 테이블을 활용한 단일 update 방식으로 개선했습니다.

결과
대용량 데이터 조회 성능 문제를 해결하고 사용자 체감 속도 및 운영 안정성을 개선했습니다.

Oracle 일반 Job 실행 중단 이슈 → DBMS_SCHEDULER 전환
Batch Oracle 안정화

상황
일정 주기로 실행되는 Oracle Job이 TOAD 환경의 일반 Job 형태로 운영되고 있었으며, 데이터 처리량 증가로 인해 실행 시간이 길어지면서 간헐적으로 Job이 중단되는 문제가 발생했습니다.

문제 분석
TOAD 일반 Job은 실행 시간이 길어질 경우 세션 및 실행 환경 제약으로 인해 안정적으로 완료되지 못하는 구조였고, 실행 중 중단 시 재시도 없이 실패 처리되는 한계가 있었습니다. 단순 로직 수정만으로는 해결이 어려운 구조적 문제였습니다.

개선 및 조치
DBMS_SCHEDULER 기반 Job이 보다 안정적인 백그라운드 실행 환경을 제공하는 것을 확인하고, 기존 Job을 DBMS_SCHEDULER.CREATE_JOB 방식으로 재구성했습니다. 세션 의존도를 낮춘 구조로 전환하여 장시간 실행에도 중단되지 않도록 개선했습니다.

결과
장시간 배치 작업에서도 Job 중단 없이 안정적으로 수행되도록 개선하였으며, 배치 실행 신뢰성과 운영 안정성을 크게 향상시켰습니다.

대량 조회로 인한 WAS 서버 다운 이슈
힙덤프파일분석 서버로그확인

상황
조회 화면에서 1개월 기준 약 25만 건 이상의 데이터가 한 번에 조회되며, 특정 시점에 WAS 서버가 다운되는 장애가 발생했습니다. 사용자 요청 직후 발생하는 고위험 운영 장애였습니다.

문제 분석
대량 데이터 조회 이후 서버가 비정상 종료되었으며, 로그만으로는 어떤 요청이 원인이었는지 추적이 어려운 상황이었습니다. Heap Dump 파일을 통해 분석한 결과, OutOfMemory 오류로 인한 JVM 메모리 부족이 원인이었습니다.

개선 및 조치
WAS 서버 내 Heap Dump 파일을 분석하여 특정 조회 요청과의 연관성을 확인하고, 대량 데이터 조회 구조가 메모리에 과부하를 주는 문제를 식별했습니다. 이후 조회 범위 제한 및 처리 방식 개선을 통해 메모리 사용 구조를 최적화하고 운영 기준을 정비했습니다.

결과
대량 조회 상황에서도 서버 다운이 재발하지 않도록 안정성을 확보했으며, 장애 발생 시 원인 추적 및 대응 체계를 개선했습니다.

Nginx 환경에서 SSL 인증서 교체 및 적용
SSL인증서 Nginx 서버설정

상황
운영 중인 서비스에서 SSL 인증서 만료 시점이 도래하여, Nginx 기반 서버 환경에서 인증서 교체 작업이 필요했습니다. 작업 실패 시 서비스 접속 불가로 이어질 수 있는 상황이었습니다.

문제 분석
인증서 파일 구성(cert, key)과 적용 방식에 대한 이해 부족 시 설정 오류 및 서비스 중단 가능성이 있었으며, 인증서 변환(OpenSSL) 및 적용 절차에 대한 문서가 부족한 상태였습니다.

개선 및 조치
SSL 인증서 구조(cert.pem, key.pem)를 명확히 이해하고, OpenSSL을 활용하여 인증서 처리 작업을 수행했습니다. 이후 Nginx 설정에 정확히 반영하고, 적용 후 서비스 정상 동작 여부를 검증했습니다.

결과
서비스 중단 없이 SSL 인증서 교체를 완료하였으며, 인증서 적용 절차에 대한 이해를 기반으로 안정적인 운영 환경을 유지했습니다.

필수 파라미터, 상태값, 처리 기준이 명확히 정의되지 않을 경우 데이터 오류 발생 가능성이 높았으며, 처리 결과 및 예외 케이스에 대한 양사 간 기준 차이가 존재했습니다.

개선 및 조치
타사 개발자와 직접 협업하여 API 목적과 데이터 흐름, 처리 시나리오를 정의하고, 요청/응답 구조 및 상태 코드를 공동 설계했습니다. Postman을 활용해 다양한 테스트 케이스를 검증하며 설계를 보완하고, 스테이징 환경에서 실제 이관 시나리오 기반 테스트를 통해 연동 안정성을 확보했습니다.

결과
API 설계 단계부터 협업하여 데이터 처리 기준을 명확히 정립하고, 안정적인 외부 시스템 연동을 구현했습니다.

오픈 시 데이터 마이그레이션 검증 및 초기 데이터 프로세스 정립
데이터마이그레이션_검증 초기데이터_세팅

상황
병원 신규 오픈 시마다 데이터 마이그레이션과 초기 데이터 세팅 작업이 반복적으로 발생했으며, 오픈 직후 정상 운영을 위해 사전 검증이 필수적인 상황이었습니다.

문제 분석
데이터 이관 및 초기 세팅이 수작업 중심으로 진행되었고, 검증 기준과 절차가 명확하지 않아 담당자마다 방식이 달라 비효율이 발생했습니다.

개선 및 조치
이관 데이터 검증 대상과 기준을 정의하고 정합성 검증 프로세스를 수립했습니다. 또한 반복적으로 필요한 초기 데이터를 정리하고, 수작업 SQL 대신 관리자 화면을 통해 데이터 입력 및 관리가 가능하도록 구조를 개선했습니다. 전체 과정을 문서화하여 표준화했습니다.

결과
데이터 마이그레이션과 초기 세팅 작업을 구조화하여 병원 오픈 시 일관된 기준으로 빠르고 안정적인 운영이 가능하도록 개선했습니다.
